ADR-014: aloop's UAC2 gadget audio path opens hw:UAC2Gadget,0 explicitly (never ALSA "default", which is ambiguous across 3 real cards and routes through dmix/dsnoop's own fixed period), with explicit hw_params (period_size=block_size exactly, buffer_size=4 periods) instead of snd_pcm_set_params' convenience 20ms-latency request, explicit sw_params start_threshold=1 period on both capture and playback (the hw_params default start_threshold for playback is the full buffer, which never got crossed since the loop only writes one period per call -- playback stayed stuck in PREPARED state forever), and the f_uac2 gadget's own req_number raised from kernel default 2 to 4 in src/usb/f_uac2-gadget.sh. All four fixes were witnessed live via SSH on a real Pi 4. A residual -EIO/frozen hw_ptr after all fixes was traced to no host application actively streaming audio through the gadget at verification time -- expected idle-USB-audio-class behavior, not a code defect.
